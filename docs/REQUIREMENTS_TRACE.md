## Remaining Proof Needed

- Railway API URL from a successful `pnpm deploy:railway:apply -- --create-domain` or equivalent `pnpm dlx @railway/cli up` deployment.
- Post-deploy `ARKSCORE_API_URL=https://... pnpm verify:railway:live` output proving the deployed Railway API is live Wavy-backed before Fuji/Vercel finalization.
- Live `/health` response from the Railway API.
- Live Wavy Node score response with `source: "wavy"` after `WAVY_NODE_API_KEY` and `WAVY_NODE_PROJECT_ID` are configured.
- Local `pnpm probe:wavy` output proving the Wavy credentials, active configured chain, production subject-hash salt, Wavy analysis id, traceability fields, subject hash, and evidence hash.
- Backend `ARKSCORE_SUBJECT_HASH_SALT` for production subject hashing.
- Fuji `CreditScoreRegistry` address from `pnpm --filter @arkscore/contracts deploy:fuji`; the deploy script writes `packages/contracts/deployments/fuji/CreditScoreRegistry.json`.
- Optional eERC20 address from the Ava Labs EncryptedERC deployment plus `pnpm probe:eerc20:strict` and `pnpm verify:live:strict:eerc20:record` output proving deployed Fuji bytecode if the privacy token demo is included.
- Local `pnpm probe:fuji` output proving the deployer key maps to a Fuji account with test AVAX.
- Authorized scorer proof from `ARKSCORE_SCORER_ADDRESS=0x... pnpm --filter @arkscore/contracts scorer:fuji` and `pnpm verify:live:strict:record`.
- Full oracle write proof from `pnpm record:fuji`, showing a live Railway Wavy response stored on Fuji and read back with matching subject hash, evidence hash, Wavy analysis id, risk score, composite score, decision, and institution.
- Vercel environment update for `NEXT_PUBLIC_API_BASE_URL` and `NEXT_PUBLIC_CREDIT_SCORE_REGISTRY_ADDRESS`, followed by a production redeploy through `pnpm finalize:live:apply`.
- Final `pnpm verify:live:strict:record` run proving the public frontend bundle contains the live Railway API URL and Fuji registry address, Railway API, live Wavy credential mode, production subject-hash salt, live Wavy `source: "wavy"` response with no-store score and rate-limit headers, Fuji registry bytecode, `hasScore(bytes32)`, `getScore(bytes32)`, authorized scorer, and the latest score-record artifact are all reachable and consistent.
